```markdown
---
name: multi-review
description: 跨模块关联审查。指定一个模块，并行 spawn 3 个 Agent 分别审查模块自身、调用者、被调用者，汇总跨模块影响面。适用于模块重构前评估波及范围。
allowed-tools: Read, Bash, Grep, Glob, Agent
---

## $ARGUMENTS

对 `$ARGUMENTS` 指定的目标模块执行三 Agent 并行审查。

---

## 步骤

### 1. 确定审查范围

从 `$ARGUMENTS` 解析目标模块路径（如 `memory/scheduler`）。如果未指定，提示用户输入。

### 2. 并行 spawn 3 个 Agent

三个 Agent 同时启动，互不依赖：

---

#### Agent 1 — 审查目标模块自身
```

你是模块内部审查专家。审查目标模块的代码质量、API 完整性和潜在缺陷。

流程：

1. 读取目标模块的所有 Python 文件
2. 列出所有公开方法/类的签名
3. 检查：
   - 每个公开方法是否有对应的调用者（不被调用的公开方法 = 死代码候选）
   - 异常处理是否完整（try/except 覆盖了所有外部调用路径）
   - 核心业务逻辑是否有明显的边界缺失（空列表、None 值、超时）
   - 模块内的内部依赖是否合理（工具函数分散在多个文件？）

输出格式：

### 目标模块：[模块名]

- **文件**: [N] 个
- **公开 API**: [N] 个方法/类

#### ⚠️ 潜在问题

- **[严重度]**: file.py:L行号 — 描述 — 原因

#### ✅ 确认无问题

- 



```
---

#### Agent 2 — 审查调用者
```

你是调用链审查专家。找出所有依赖目标模块的模块，检查调用方式是否正确。

流程：

1. Grep 全项目搜索 `from 目标模块` 和 `import 目标模块`
2. 列出所有调用者及其调用点（文件:行号 + 调用的具体方法）
3. 对每个调用点检查：
   - 调用签名与目标方法定义是否匹配
   - 调用者是否正确处理了目标方法的返回值
   - 调用者是否有异常保护（如果目标方法可能抛异常）
   - 是否有循环依赖风险（目标模块又 import 了调用者）

输出格式：

### 调用者清单（[N] 个）

#### 调用者: module/file.py

- **调用点**: L行号 — 调了 `方法名()`
- **签名匹配**: ✅ / ⚠️ — [说明]
- **返回值处理**: ✅ / ⚠️ — [说明]
- **循环依赖风险**: 有 / 无

### 汇总

- 总调用点: [N]
- ⚠️ 风险项: [N]



```
---

#### Agent 3 — 审查被调用者
```

你是依赖关系审查专家。列出目标模块依赖的所有模块，检查依赖是否合理。

流程：

1. 读取目标模块的 import 语句，列出所有依赖模块
2. 对照 CLAUDE.md 依赖拓扑，检查每个依赖是否符合声明
3. 对每个依赖检查：
   - 该依赖是必要的还是冗余的（目标模块到底用没用这个依赖）
   - 依赖方向是否符合拓扑分层（低层不应依赖高层）
   - 同一被调用者的方法在目标模块中是否有一致的使用模式

输出格式：

### 依赖清单（[N] 个）

| 依赖模块 | 使用的方法         | 拓扑合规 | 备注         |
| -------- | ------------------ | :------: | ------------ |
| module_x | method_a, method_b |    ✅     | —            |
| module_y | 仅 TYPE_CHECKING   |    ✅     | 运行时无依赖 |

#### ⚠️ 异常依赖

- file.py:L行号 — `from module_z import ...` — 拓扑未声明
- file.py:L行号 — `import module_w` — 低层依赖高层，违反分层

### 汇总

- 总依赖: [N]
- ⚠️ 风险项: [N]



```
---

### 3. 汇总输出

三个 Agent 完成后，汇总为：

## 跨模块审查报告：`[目标模块]`

### 影响面一览
| 维度 | 数据 | ⚠️ 风险 |
|------|------|:--:|
| 模块自身 | [N] 公开 API | [N] |
| 调用者 | [N] 个模块，[N] 个调用点 | [N] |
| 被调用者 | [N] 个依赖 | [N] |

### 高风险项（必须处理）
[汇总三个 Agent 中严重度为"高"的项]

### 中风险项（建议处理）
[汇总严重度为"中"的项]

### 修改建议
- 如果要改 [目标模块]，建议重点关注 [调用者1]、[调用者2]
- 如果改变了 [方法X] 的签名，需要同步更新 [N] 个调用点
- 如果不确定影响面，敲 `/doc-audit` 确认文档拓扑

---

## 边界情况
- 目标模块不存在: 提示用户检查路径，退出
- 目标模块无调用者: 注明，可能是死代码候选
- 目标模块无被调用者: 注明，可能是顶层入口
- 非 Python 项目: 报告并退出
```
